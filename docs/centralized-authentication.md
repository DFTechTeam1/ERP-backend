# Centralized Authentication (RS256)

Laravel (`erp-backend`) is the **sole issuer** of access tokens for the whole
ERP. It signs short-lived RS256 JWTs; every other service verifies them
*locally* with the public key ŌĆö no per-request callback to Laravel.

| Service            | Role     | Key it needs              | How it gets it            |
|--------------------|----------|---------------------------|---------------------------|
| `erp-backend`      | Issuer   | private **+** public      | `JWT_PRIVATE_KEY` / `JWT_PUBLIC_KEY` env |
| `erp-backend-node` | Consumer | public only               | `JWT_PUBLIC_KEY` env      |
| `erp-frontend`     | Consumer | none                      | holds the token, sends it as `Authorization: Bearer` |

---

## ŌÜĀ’ĖÅ Two different keypairs ŌĆö do not confuse them

This repo has **two independent RSA keypairs** for two unrelated features:

| Keypair | Used for | Where it lives | Generated by |
|---------|----------|----------------|--------------|
| **Centralized auth** (this doc) | Login / session access tokens | `JWT_PRIVATE_KEY`, `JWT_PUBLIC_KEY` env vars (base64 PEM) | `php artisan jwt:keygen` |
| **MCP OAuth 2.1** | MCP bearer tokens + `/.well-known/jwks.json` | `storage/oauth/private.key`, `storage/oauth/public.key` files | `make oauth-keys` / `erp_infra/secrets/oauth/` |

Installing the OAuth keys (`make oauth-keys`) does **not** fix login, and vice
versa. The `Key cannot be empty` error below is always the *centralized-auth*
keypair (`JWT_PRIVATE_KEY`), never the OAuth files.

## The `Key cannot be empty` error

```
message:   "Key cannot be empty"
exception: "Lcobucci\JWT\Signer\InvalidKeyProvided"
file:      vendor/lcobucci/jwt/src/Signer/InvalidKeyProvided.php:39
```

**Cause.** `config/jwt.php` reads the signing key from `JWT_PRIVATE_KEY`. When
that env var is unset, `config('jwt.private_key')` is `null`, and
[`TokenService`](../app/Services/Auth/TokenService.php) hands an empty string to
the lcobucci signer at login time ŌåÆ it throws.

**Fix.** Generate a keypair, set the two env vars, reload config. See below.

## Generate the keypair

A dedicated Artisan command lives at
[`JwtGenerateKeys`](../app/Console/Commands/JwtGenerateKeys.php):

```sh
# inside the erp-backend container (or via the infra Makefile):
php artisan jwt:keygen
#   from erp_infra/:  make artisan cmd="jwt:keygen"
```

It prints two base64-encoded PEM lines (base64 so the multi-line PEM survives
`.env` parsing and Docker env injection):

```
JWT_PRIVATE_KEY=LS0tLS1CRUdJTiBQUklWQVRF...
JWT_PUBLIC_KEY=LS0tLS1CRUdJTiBQVUJMSUM...
```

Options:

| Flag | Effect |
|------|--------|
| `--bits=2048` | RSA key size (2048 is the enforced minimum). |
| `--write`     | Also write the raw PEM pair to `storage/jwt/` (gitignored). |
| `--force`     | Overwrite existing `storage/jwt` files when used with `--write`. |

> **One keypair per environment.** Never reuse the same keys across
> dev / staging / production. A leaked dev key must not be able to mint
> production-valid tokens.

## Install the keys

### Dockerized setup (this project)

Both the `laravel` and `express` services load `erp_infra/.env` via
`env_file:`. Paste **both** lines into `erp_infra/.env`:

```dotenv
# ---- Centralized auth (RS256 access tokens) ----
JWT_PRIVATE_KEY=...   # laravel only
JWT_PUBLIC_KEY=...    # laravel + express both read this
```

`express` simply ignores the private key. Then recreate the containers so they
re-read the env file (a plain `restart` does **not** pick up `env_file`
changes ŌĆö the container must be recreated):

```sh
cd erp_infra
docker compose --env-file .env -f compose.yaml -f compose.dev.yaml up -d --force-recreate laravel express
make artisan cmd="config:clear"   # only needed if config was cached
```

Verify the key is loaded:

```sh
make artisan cmd="tinker --execute='echo strlen((string)config(\"jwt.private_key\"));'"
# -> 1704 (non-zero). 0 means the env var still isn't reaching the container.
```

> **Gotcha ŌĆö `environment:` overrides `env_file:`.** If a value is *also*
> hardcoded under a service's `environment:` block in `compose.yaml`, that wins
> over `.env`. (This is exactly what bit the CORS vars: they were pinned to
> `"*"` in `compose.yaml` and ignored the `.env`.) Keep auth/CORS values driven
> from `.env` via `${VAR:-default}` substitution, not hardcoded.

### Bare-metal setup

Put the same two lines in `app/erp-backend/.env` (Laravel) and the public key in
the Node service's `.env`, then `php artisan config:clear`.

## How it works

### Login ŌåÆ access token + refresh cookie

`POST /api/auth/login` ŌåÆ [`LoginController`](../app/Http/Controllers/Api/Auth/LoginController.php):

1. Validate credentials.
2. [`TokenService::issueAccessToken()`](../app/Services/Auth/TokenService.php)
   signs an RS256 JWT (header `alg=RS256`, `kid=key-1`). Claims:
   - `iss` = `JWT_ISSUER`, `aud` = `JWT_AUDIENCE`, `sub` = user id, `jti` = uuid
   - `roles` (array) and `permissions` (flat array of names) ŌĆö so consumers
     authorize **locally** without calling back.
   - It deliberately carries **no** menu and **no** secret/PII ŌĆö the payload is
     readable by the user. The menu is served fresh from `GET /api/menu`.
3. [`RefreshTokenService`](../app/Services/Auth/RefreshTokenService.php) issues a
   rotating opaque refresh token, returned as an **httpOnly cookie**.

The access token is returned in the JSON body; the frontend keeps it in memory
and sends it as `Authorization: Bearer <token>` on every request.

### Refresh rotation (theft detection)

`POST /api/auth/refresh` reads the httpOnly cookie and calls
`RefreshTokenService::rotate()`:

- Each refresh token belongs to a **family**. Using one issues a new token and
  revokes the old one.
- Replaying an already-rotated token outside the grace window
  (`JWT_REFRESH_GRACE_SECONDS`, default 10s) is treated as **theft** ŌåÆ the whole
  family is revoked. Inside the grace window it's treated as a benign race
  (e.g. two tabs refreshing at once).
- Expired/revoked rows are pruned daily at 02:00 by the scheduled
  `PruneRefreshTokens` command (`JWT_REFRESH_PRUNE_AFTER`, default 7 days).

### Verification by consumers

- **Laravel** protects routes with the `jwt.auth` middleware
  ([`AuthenticateWithAccessToken`](../app/Http/Middleware/AuthenticateWithAccessToken.php)),
  aliased in `bootstrap/app.php`.
- **Express** (`erp-backend-node`) verifies in `src/core/middleware/bearerToken.ts`
  using `JWT_PUBLIC_KEY` (base64 PEM ŌåÆ `importSPKI`, `algorithms: ['RS256']`),
  maps the token `sub` to the Laravel user id. No network call to Laravel.

## Configuration reference (`config/jwt.php`)

| Env var | Default | Meaning |
|---------|---------|---------|
| `JWT_PRIVATE_KEY` | ŌĆö | base64 PEM RSA private key (Laravel signs). **Required.** |
| `JWT_PUBLIC_KEY`  | ŌĆö | base64 PEM RSA public key (all verifiers). **Required.** |
| `JWT_ISSUER`      | `https://auth.dfactory.pro` | `iss` claim. |
| `JWT_AUDIENCE`    | `erp` | `aud` claim. |
| `JWT_KID`         | `key-1` | `kid` header for future JWKS rotation. |
| `JWT_ACCESS_TTL`  | `30` | Access-token lifetime, **minutes**. |
| `JWT_REFRESH_TTL` | `1` | Refresh lifetime, **days**. |
| `JWT_REFRESH_TTL_REMEMBER` | `30` | Refresh lifetime with "remember me", days. |
| `JWT_REFRESH_GRACE_SECONDS` | `10` | Replay grace window. |
| `JWT_REFRESH_PRUNE_AFTER` | `7` | Prune revoked/expired rows after N days. |
| `JWT_REFRESH_COOKIE` | `df_refresh_token` | Cookie name. |
| `JWT_REFRESH_COOKIE_DOMAIN` | ŌĆö | Cookie domain (e.g. `.dfactory.pro`). |
| `JWT_REFRESH_COOKIE_PATH` | `/` | Cookie path. |
| `JWT_REFRESH_COOKIE_SECURE` | `true` | `Secure` flag. |
| `JWT_REFRESH_COOKIE_SAME_SITE` | `lax` | `SameSite` policy. |

### Endpoints

| Method | Path | Auth | Purpose |
|--------|------|------|---------|
| POST | `/api/auth/login` | public | credentials ŌåÆ access token + refresh cookie |
| POST | `/api/auth/refresh` | refresh cookie | rotate refresh, return new access token |
| POST | `/api/auth/logout` | `auth.session` | revoke refresh, clear cookie |
| GET  | `/api/menu` | `jwt.auth` | menu tree for the authenticated user |
| POST | `/api/internal/auth/service-token` | `internal.service` (HMAC) | mint a centralized token for the service account |

### Service-to-service tokens (`/api/internal/auth/service-token`)

Internal services (e.g. erp-backend-node) sometimes need to call other services
that verify the centralized token, but have no end-user token in context (e.g.
background jobs). Rather than forwarding a user token or storing a password,
they POST to this endpoint, authenticated by the **internal-service HMAC**
(`INTERNAL_SERVICE_SECRET`, the same mechanism as `/api/internal/*`). It mints an
access token via `TokenService` for the user configured in
`JWT_SERVICE_ACCOUNT_EMAIL` and returns `{ data: { access_token } }`.

The identity is **fixed server-side** by config ŌĆö the caller cannot request a
token for an arbitrary user ŌĆö so a leaked request body can't escalate privileges.
Set `JWT_SERVICE_ACCOUNT_EMAIL` to a real, low-privilege ERP user.
(`ServiceTokenController`; consumed by erp-backend-node's `getServiceAccessToken`.)

## Refresh-cookie behavior in local dev (http)

The refresh cookie defaults to `Secure=true; SameSite=lax`. With those defaults
the **refresh** flow returns `Unauthenticated` on every call, because the cookie
never reaches the server (`$request->cookie('df_refresh_token')` is `null`).
Login itself still works ŌĆö it returns the access token in the body ŌĆö so only
refresh is affected.

Why the cookie is dropped in the default dev setup:

- The frontend (`erp.localhost:8080` / `localhost:5173`) and the API
  (`backend.localhost:8080`) are **cross-site**. "Same-site" is decided by the
  registrable domain (eTLD+1); since `localhost` is the eTLD, `erp.localhost`
  and `backend.localhost` are *different* sites. A `SameSite=Lax` cookie is
  **never** attached to cross-site XHR/fetch ŌĆö only to top-level navigations ŌĆö
  so it is omitted from the refresh request.
- A cross-site cookie therefore needs `SameSite=None`, and `SameSite=None`
  **requires `Secure`**. Over http that is normally rejected, but browsers treat
  any host ending in `.localhost` as a **secure context**, so the Secure cookie
  is still accepted in dev.

**Fix for the cross-host dev setup (current architecture):**

```dotenv
JWT_REFRESH_COOKIE_SECURE=true
JWT_REFRESH_COOKIE_SAME_SITE=none
```

Do **not** use `SameSite=lax` + `Secure=false` here ŌĆö Lax is not sent cross-site,
so refresh would still fail. `lax`/`false` is only correct if you serve the SPA
**same-origin** with the API (same host:port, e.g. both behind the edge nginx),
in which case the request is same-origin and Lax is satisfied with no Secure.

In **staging/prod** keep `Secure=true`; for a cross-site SPA use
`SameSite=None` + `Secure=true` + a shared parent domain
(`JWT_REFRESH_COOKIE_DOMAIN=.dfactory.pro`).

> Cross-origin login also requires CORS to allow credentials ŌĆö see the matching
> `CORS_ALLOWED_ORIGINS` (explicit origins, not `*`) and
> `CORS_SUPPORTS_CREDENTIALS=true` in `config/cors.php` / `erp_infra/.env`.

## Key rotation

1. `php artisan jwt:keygen` to mint a new pair; bump `JWT_KID` (e.g. `key-2`).
2. Roll out the new `JWT_PUBLIC_KEY` to **all verifiers first** (Laravel +
   Express) so in-flight tokens stay valid.
3. Switch `JWT_PRIVATE_KEY` on Laravel to start signing with the new key.
4. After the old access-token TTL (`JWT_ACCESS_TTL`, ~30 min) has fully elapsed,
   retire the old public key.

Because access tokens are short-lived, a full rotation drains in well under an
hour with no forced logout.

## Troubleshooting

| Symptom | Likely cause |
|---------|--------------|
| `Key cannot be empty` on login | `JWT_PRIVATE_KEY` unset / empty in the issuer's env. |
| `priv_len=0` from the tinker check | env not reaching the container ŌĆö recreate (not restart); check for an `environment:` override in `compose.yaml`. |
| Express returns 401 on a valid token | `JWT_PUBLIC_KEY` missing/mismatched on the Node side, or keys regenerated on Laravel only. |
| `iss`/`aud` validation fails | `JWT_ISSUER` / `JWT_AUDIENCE` differ between issuer and verifier. |
| Login works but session drops after ~30 min | refresh cookie not stored/sent ŌĆö see the dev-cookie section above. |
| Config changes ignored | config cached ŌĆö run `php artisan config:clear`. |
